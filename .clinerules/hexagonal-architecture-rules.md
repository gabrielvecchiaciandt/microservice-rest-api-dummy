## Architecture Principles

### Layered Architecture

The application follows a clean architecture approach with strict layer separation:

**Core Layer** (Framework-independent):
- `core.entity` - Domain entities
- `core.usecase` - Business logic
- `core.dataprovider` - Data provider interfaces
- `core.usecase.step` - Use case steps
- `core.usecase.strategies` - Business strategies

**Adapter Layers**:
- `dataproviders` - External data source implementations
- `entrypoints` - API controllers and endpoints
- `configuration` - Dependency injection configuration

**Generated Code** (DO NOT MODIFY):
- `build/generated/src/main/java/br/com/alelo/api`
- `build/generated/src/main/java/br/com/alelo/dto`

### Dependency Rules

- Core layer MUST NOT access other layers or frameworks
- Core layer MUST NOT use Spring annotations
- `dataproviders` and `entrypoints` accessed only via `configuration`
- `configuration` layer CANNOT be accessed by any other layer
- Exception: `br.com.flowtechsolutions.observability.application.logger` is allowed in Core layer

### Dependency Injection

- Dependencies injected ONLY via configuration beans
- DO NOT use Spring's automatic component scanning
- Example:
  ```java
  @Configuration
  class BusinessUseCaseConfiguration {
      @Bean
      public BusinessUseCase businessUseCase(DatabaseProvider databaseProvider) {
          return new BusinessUseCase(databaseProvider);
      }
  }
  ```

## Code Quality Standards

### General Principles

- **SOLID Principles**: Adhere strictly to SOLID principles in every Java class
- **Clean Code**: Write self-documenting code with meaningful names
- **Single Responsibility**: Functions should do one thing
- **No Duplication**: Avoid code duplication
- **Conciseness**: Keep code concise and focused on solutions over explanations

### Code Style

- Follow Google Java Style Guide conventions
- Import classes explicitly (NO wildcard `*` imports)
- Use meaningful variable names in English
- Prefer composition over inheritance
- Maximum 7 parameters per method (use DTOs for parameter grouping)
- Maximum cognitive complexity of 15 per method

### DTOs and Records

- Use record classes for DTOs and immutable data models
- Example:
  ```java
  public record UserDTO(String name, String email, LocalDate birthDate) {}
  ```

### Constants and Entities

- Place constants in `core/entity/constants`
- Map database entities in `dataproviders/database/entity`

### Documentation

- Include Javadoc for public methods and classes
- Document non-obvious implementation details
- Use springdoc-openapi annotations on controllers for comprehensive Swagger documentation
- DO NOT add unnecessary comments in code (code should be self-explanatory)

### Error Handling

- Use specific exceptions over generic ones
- Implement proper exception handling with meaningful error messages
- Centralize exception logging in `ApplicationExceptionHandler`
- AVOID redundant logs in controllers, repositories, and simple use cases

## API Development (API-First Approach)

### Creating New Endpoints

1. **OpenAPI Specification**
    - Use `openApiSpec.yaml` created by Sensedia
    - NEVER modify `openApiSpec.yaml` manually
    - Generated classes in `api` and `dto` are immutable

2. **Controller Implementation**
    - Create in `java/br/com/alelo/plataformavendas/entrypoints/rest`
    - Implement the interface generated by OpenAPI
    - Example:
      ```java
      @RestController
      public class ProductsController implements ProductsApi {
          private final GetAllProductsUseCase getAllProductsUseCase;
          
          public ProductsController(GetAllProductsUseCase getAllProductsUseCase) {
              this.getAllProductsUseCase = getAllProductsUseCase;
          }
          
          @Override
          public ResponseEntity<ProductResponse> productsGet() {
              ProductResponse response = ProductMapper.toProductResponseData(
                  getAllProductsUseCase.execute()
              );
              return ResponseEntity
                  .ok()
                  .header("Content-Type", "application/json")
                  .body(response);
          }
      }
      ```

3. **Mapper Creation**
    - Create in `java/br/com/alelo/plataformavendas/entrypoints/rest/mapper`
    - Convert domain classes (core) to DTOs generated by OpenAPI

4. **Integration**
    - Controller calls only the corresponding use case
    - All dependencies injected via configuration

## Logging Standards

### Framework

- MUST use SLF4J as logging facade with Logback implementation
- Import ONLY `org.slf4j.Logger` and `org.slf4j.LoggerFactory`
- NEVER import `java.util.logging`, Log4j, or commons-logging

### Logger Declaration

- Declare as: `private static final Logger logger = LoggerFactory.getLogger(ClassName.class);`
- NEVER use string literals for logger names
- NEVER create logger instances in methods or constructors

### Log Levels

- **ERROR**: Critical failures requiring immediate attention
- **WARN**: Unexpected situations not preventing main functionality
- **INFO**: Normal system events documenting application flow
- **DEBUG**: Detailed debugging information (disabled in production)
- **TRACE**: Extremely detailed information (rarely used)

### Sensitive Data Protection

NEVER log sensitive data including:
- PII: CPF, RG, names, addresses, birth dates, emails, phone numbers
- Financial: credit card numbers, CVV, bank details, transaction amounts
- Credentials: passwords, tokens, API keys, JWT tokens, session cookies
- Biometric and medical data
- NEVER log complete objects that may contain sensitive data

### Best Practices

- Use placeholders `{}` instead of string concatenation:
  ```java
  logger.debug("Processing user: {}", userId);
  ```
- Create descriptive, contextualized messages with sufficient debugging information
- Implement masking functions for potentially sensitive data
- Use MDC (Mapped Diagnostic Context) for correlation IDs
- Configure JSON format for production logs using LogstashEncoder
- Configure log rotation using RollingFileAppender
- Centralize exception logging in `ApplicationExceptionHandler`

### Audit Logging

- Create dedicated audit logger for security-sensitive actions:
  ```java
  private static final Logger auditLogger = LoggerFactory.getLogger("AUDIT");
  ```
- Log: who, what, when, where, and status
- Configure separate file with extended retention

### Centralized Exception Logging

- SHOULD centralize exception logging in `ApplicationExceptionHandler`
- AVOID redundant logs in controllers, repositories, and simple use cases
- ONLY add logs in business layers for:
    - Critical performance operations (execution time > threshold)
    - Security-sensitive audit actions
    - Complex business logic debugging (DEBUG level)
- NEVER duplicate exception logs already handled centrally

## Testing Standards

### Framework and Tools

- Use JUnit Jupiter for test framework
- Use AssertJ for assertions (especially for Collections)
- Use Instancio library for creating mock objects
- Use Mockito for mocking (with ArgumentCaptors and verify when appropriate)

### Test Naming

- Pattern: `should_somethingInCamelCase_when_somethingInCamelCase`
- Use `@DisplayName` with clear descriptions in Brazilian Portuguese (pt_BR)
- Example:
  ```java
  @DisplayName("Deve retornar o enum correto para cada tipo de produto.")
  @Test
  void should_returnCorrectEnum_when_validProductType() {
      // test implementation
  }
  ```

### Test Implementation

- Create separate mocks for request and response objects
- Follow good practices for concise and clean mock objects
- DO NOT create unit tests for non-public class methods
- Simulate mutation tests to ensure adequate coverage
- Use `assertThrows()` for testing error and exception scenarios

### Using Instancio

- For specific field values in DTO classes:
  ```java
  Object object = Instancio.of(Object.class)
      .set(field(Object::getField), "value");
  ```
- For specific field values in records:
  ```java
  Object object = Instancio.of(Object.class)
      .set(field(Object::field), "value");
  ```

## Git Workflow

### Commit Message Structure

Format:
```
[TYPE] #taskNumber: title

description
```

- Messages MUST be in Portuguese
- Include brackets `[]` around TYPE
- Leave blank line between title and description
- Title must be short, objective, and concise
- Description must provide implementation details
- Each commit should focus on a group of related files

### Commit Types

- **feat**: nova funcionalidade para o usuário
- **fix**: correção de bug para o usuário
- **docs**: mudanças na documentação
- **style**: mudanças que não afetam o significado do código
- **refactor**: mudança no código que não corrige um bug nem adiciona funcionalidade
- **test**: adição ou correção de testes
- **chore**: mudanças na build ou em arquivos auxiliares e bibliotecas
- **ci**: mudanças de CI/CD
- **perf**: melhora de performance

### Commit Rules

- DO NOT commit without asking if the user wants to commit the changes
- Each commit should focus on related changes by type
- Ensure blank line between title and description
